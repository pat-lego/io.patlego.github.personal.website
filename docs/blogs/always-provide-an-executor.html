Executors: Asynchronous processing and defaults

Developing with asynchronous processes in Java is a powerful feature, but understanding what it is doing behind the scenes is something we quickly forget. When we develop using CompletableFuture, we provide the necessary lambda function to execute in the background.

The question is: what happens to that function after it starts to execute? The answer is Executors. When we run the lambda asynchronously without a provided executor, we run the lambda in a common pool (also known as the ForkJoinPool.commonPool()) which has a default size of # of processors - 1. This means that if you have 10 cores then you would be able to run 9 threads in parallel. The common pool cannot be changed or mutated — it is provided by the JVM for shared usage.

Now here is a hypothetical: let's say a burst of lambda functions run saturating the Common Pool. That would not normally be seen as a bad thing, unless those lambdas are waiting for a response from a separate lambda that also runs in the common pool. With this we have exposed a deadlock with asynchronous processing. All threads in the common pool are waiting for threads to provide a result, but the threads that are to provide the result also run in the common pool and will never receive CPU time to do so. This can be seen in the thread dump below, where we have 9 Common Pool threads that are BLOCKED, preventing any other threads from being executed in the Common Pool.

How do we prevent this from happening? Provide an Executor to avoid using the Common Pool. We can leverage the following API. This allows threads to run within their own pool, preventing them from consuming slots in the Common Pool.

❌ Without an Executor — runs in the Common Pool (dangerous)

/ No executor provided — CompletableFuture uses ForkJoinPool.commonPool() // All tasks share the same fixed-size pool (# processors - 1 threads) // If these tasks wait on each other, the pool deadlocks CompletableFuture<String> outer = CompletableFuture.supplyAsync(() -> { // This task runs in the Common Pool and then waits for inner to finish CompletableFuture<String> inner = CompletableFuture.supplyAsync(() -> { // This also runs in the Common Pool — but all slots may already be taken return fetchDataFromElastic(); }); return inner.join(); // DEADLOCK: outer holds a pool thread waiting for inner, // but inner can never get a pool thread to run });

✅ With a dedicated Executor — isolated from the Common Pool (safe)

// Dedicated thread pool — completely separate from ForkJoinPool.commonPool() // Size it based on your workload: I/O-bound tasks can use more threads than CPU cores ExecutorService executor = Executors.newFixedThreadPool(10); CompletableFuture<String> outer = CompletableFuture.supplyAsync(() -> { // Runs in the dedicated executor, not the Common Pool CompletableFuture<String> inner = CompletableFuture.supplyAsync(() -> { // Also runs in the dedicated executor — has its own available threads return fetchDataFromElastic(); }, executor); return inner.join(); // Safe: inner can always acquire a thread from the pool }, executor); // Always shut down the executor when the component is deactivated executor.shutdown();
