Query Abuse in Sling Models: How Per-Component JCR Queries Create I/O Contention

A Sling Model that runs a JCR query in its @PostConstruct looks harmless in isolation. It takes a few milliseconds, returns a handful of nodes, and the page renders cleanly in your local environment. Under production load, with dozens of components on a single page all firing their own queries simultaneously, it turns into a wall of threads blocked on a kernel-level I/O lock. This post follows a real thread dump to show exactly how that happens and what to do instead.

The thread dump

The following stack trace was captured from a production AEM instance. Thread #97 had been alive for over 43 minutes and was completely blocked — not doing useful work, just waiting for a lock:

"192.0.2.1 [1779799224675] GET /content/mysite/markets/region/en/about/our-vision.html" #97 prio=5 java.lang.Thread.State: BLOCKED (on object monitor) at sun.nio.ch.NativeThreadSet.add(NativeThreadSet.java:50) - waiting to lock <0x00000005a9063bd8> (a sun.nio.ch.NativeThreadSet) at sun.nio.ch.FileChannelImpl.readInternal(FileChannelImpl.java:976) at sun.nio.ch.FileChannelImpl.read(FileChannelImpl.java:965) at org.apache.lucene.store.NIOFSDirectory$NIOFSIndexInput.readInternal(...) at org.apache.lucene.codecs.BlockTreeTermsReader$FieldReader$SegmentTermsEnum.seekExact(...) at org.apache.lucene.search.TermQuery.createWeight(TermQuery.java:174) at org.apache.lucene.search.IndexSearcher.search(IndexSearcher.java:289) at org.apache.jackrabbit.oak.plugins.index.lucene.LucenePropertyIndex$1.loadDocs(...) at org.apache.jackrabbit.oak.query.QueryImpl$RowIterator.hasNext(QueryImpl.java:960) at org.apache.jackrabbit.oak.jcr.query.QueryResultImpl.getNodes(QueryResultImpl.java:176) at com.day.cq.wcm.msm.impl.StatusUtil.hasChildOfTypesQuery(StatusUtil.java:280) at com.day.cq.wcm.msm.impl.LiveRelationshipManagerImpl$LiveRelationshipIterator.next(...) at com.mysite.core.models.utils.LocalizationUtils.getLiveCopiesPaths(LocalizationUtils.java:30) at com.mysite.core.models.internal.v1.ExperienceFragmentModelImpl.findMatchingLocalizedFragmentPath(...) at com.mysite.core.models.internal.v1.ExperienceFragmentModelImpl.getName(ExperienceFragmentModelImpl.java:82)

Reading from the bottom up: a Sling Model method — ExperienceFragmentModelImpl.getName() — triggers a call to LocalizationUtils.getLiveCopiesPaths(), which runs a JCR query via the AEM Multi-Site Manager live relationship API. That query is backed by Oak's Lucene property index, and the index data must be read from disk. The thread then blocks while waiting to acquire a lock on a sun.nio.ch.NativeThreadSet.

Where the Lucene index lives

Unlike JCR node properties — which are stored inside the Oak repository's NodeStore — Lucene indexes in AEM are serialized as binary segment files on the local filesystem under a path such as crx-quickstart/repository/index/. These are standard Lucene .cfs, .fdt, and .tim files — completely external to the JCR content tree. Oak's LucenePropertyIndex serves as the bridge: it translates an XPath or SQL-2 query into a Lucene IndexSearcher operation and reads the result from those files directly.

This matters because it means every JCR query that hits the Lucene index eventually becomes a file I/O operation. There is no in-memory shortcut for a cold or partially-warm OS page cache. If the relevant index segments have not been read recently enough to remain in the buffer cache, the JVM must wait on a real disk read.

What sun.nio.ch.FileChannelImpl is

Java reads those Lucene segment files through sun.nio.ch.FileChannelImpl, the internal implementation of java.nio.channels.FileChannel. Unlike the old java.io.FileInputStream, FileChannel uses kernel-level scatter/gather I/O primitives (specifically pread(2) on Linux) to transfer data directly between the kernel's page cache and the JVM heap, bypassing a user-space copy. This is generally faster — but the implementation must register the calling thread in a NativeThreadSet before each read so that the JVM can interrupt the native call cleanly if the channel is closed.

The NativeThreadSet is protected by a single Java monitor (the - waiting to lock <0x00000005a9063bd8> line in the dump). Every thread performing a read on the same FileChannel must acquire that monitor in order to register itself, do the read, and deregister. That serializes entry to the read path across all threads sharing the channel.

Why a burst of queries creates contention

A modern AEM page template commonly contains between ten and thirty components. If each component is backed by a Sling Model that independently issues a JCR query in its @PostConstruct, a single page render becomes a burst of ten to thirty simultaneous Lucene reads. Every one of those threads races to enter NativeThreadSet.add(). Only one succeeds at a time. The rest queue up in BLOCKED state waiting for the monitor, as shown in the dump above.

Now scale that to ten concurrent page requests. You have three hundred threads contending on the same lock, each holding a request thread from the Sling request processing pool. The pool drains, new requests queue up, response times climb into the seconds, and the system begins returning 503 or 502 errors. The root cause is not a slow query — it is the cumulative I/O pressure created by running the same logical operation dozens of times per render instead of once.

The situation is amplified by OS-level constraints. The Linux kernel enforces limits on the number of open file descriptors per process (ulimit -n), the maximum number of memory-mapped regions (vm.max_map_count), and the total I/O bandwidth available to the JVM process. A Lucene NIOFSDirectory holds file descriptors open for each active segment reader. As the number of concurrent queries grows, so does the descriptor count. Hit the limit and the JVM starts throwing java.io.IOException: Too many open files, which brings Lucene down entirely and prevents any content query from completing.

The dangerous pattern

The stack trace traces back to this logical structure — a query that is perfectly reasonable in isolation but catastrophic when it runs once per component per request:

❌ Query executed once per component per request (dangerous)

@Model(adaptables = SlingHttpServletRequest.class) public class ExperienceFragmentModelImpl { @SlingObject private ResourceResolver resourceResolver; private String localizedPath; @PostConstruct protected void init() { // Runs a JCR query — every instance of this component // on the page triggers a separate Lucene I/O burst List<String> liveCopies = LocalizationUtils.getLiveCopiesPaths( resourceResolver, getCurrentPath() ); this.localizedPath = findMatch(liveCopies); } }

If a page has eight Experience Fragment components, this query executes eight times in parallel for every request. Every execution is an independent Lucene search, an independent call into FileChannelImpl.read(), and an independent contender on the NativeThreadSet lock. The query result is identical every time — the live copy structure does not change between component renders on the same request — but the I/O cost is paid eight times anyway.

The fix: run the query once per render

The live copy paths for a given page are the same regardless of which component asks for them. That makes them a perfect candidate for request-scoped caching. There are two clean ways to accomplish this in AEM.

The first approach is to move the query into a higher-level Sling Model that is adapted once per request and injected into each component model. Because Sling can cache model adaptations within a request, the query runs exactly once:

✅ Query result computed once and shared across components (safe)

@Model(adaptables = SlingHttpServletRequest.class, defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL) public class LocalizationContextModel { @SlingObject private ResourceResolver resourceResolver; @Self private SlingHttpServletRequest request; private List<String> liveCopiesPaths; @PostConstruct protected void init() { // Executes once per request regardless of how many // components inject this model this.liveCopiesPaths = LocalizationUtils.getLiveCopiesPaths( resourceResolver, request.getResource().getPath() ); } public List<String> getLiveCopiesPaths() { return liveCopiesPaths; } } // Component model — no query, just a lookup @Model(adaptables = SlingHttpServletRequest.class) public class ExperienceFragmentModelImpl { @Self private SlingHttpServletRequest request; private String localizedPath; @PostConstruct protected void init() { // Reuses the cached result — zero additional I/O LocalizationContextModel ctx = request.adaptTo(LocalizationContextModel.class); if (ctx != null) { this.localizedPath = findMatch(ctx.getLiveCopiesPaths()); } } }

The second approach is to store the result directly in a request attribute, which acts as a plain key-value cache scoped to the current HTTP request:

public class LocalizationUtils { private static final String ATTR_LIVE_COPIES = LocalizationUtils.class.getName() + ".liveCopies"; @SuppressWarnings("unchecked") public static List<String> getLiveCopiesPaths( SlingHttpServletRequest request, ResourceResolver resolver, String pagePath) { List<String> cached = (List<String>) request.getAttribute(ATTR_LIVE_COPIES); if (cached != null) { return cached; } // First caller on this request pays the I/O cost List<String> result = queryLiveCopies(resolver, pagePath); request.setAttribute(ATTR_LIVE_COPIES, result); return result; } }

Both patterns produce the same outcome: one Lucene read per page request, regardless of how many components need the data. The I/O contention disappears because the NativeThreadSet lock is acquired once, not once per component.

The broader principle

JCR queries are not free. Each one that hits the Lucene index reads from a binary file on disk through a kernel I/O path that is subject to file descriptor limits, OS page cache pressure, and a shared Java monitor that serializes concurrent readers. A single query is rarely the problem. The problem is multiplying that cost by the number of components on a page and then again by the number of concurrent requests hitting the server.

Before adding a query to a Sling Model, ask whether the same data will be needed by other components on the same page during the same request. If yes, fetch it once at the request boundary — in a shared model, a request attribute, or a servlet filter — and let every component read from that in-memory result. The Lucene index and the OS I/O subsystem are shared resources. Treating them that way in your code is what keeps thread dumps boring.
